# Дизайн: Автоматическое создание операций из буфера обмена

## Дата: 2026-05-19

## 1. Обзор

Автоматическое создание операций при активации приложения на основе текста из буфера обмена. Пользователю показывается диалог с распознанными данными, после подтверждения открывается форма операции с предзаполненными полями.

**Источник операции:** `source = TransactionSource.CLIPBOARD`, `sourceData =` оригинальный текст из буфера

## 2. Структура данных

### 2.1 Расширение таблицы Transaction

| Поле | Тип | Описание |
|------|-----|----------|
| `source` | TransactionSource | Источник операции (MANUAL, SMS, PUSH, **CLIPBOARD**) |
| `sourceData` | String? | Оригинальный текст из буфера |
| `shop` | String? | Магазин из сообщения (парсится из именованной группы `shop`) |

### 2.2 Расширение таблицы accounts

| Поле | Тип | Описание |
|------|-----|----------|
| `autoCreateEnabled` | Boolean | Включить авто-создание для счёта |
| `cardMask` | String | Последние 4 цифры карты (например `1026`) |
| `regexForText` | String | Regex для парсинга текста сообщения |

### 2.2 SharedPreferences

| Ключ | Тип | Описание |
|------|-----|----------|
| `clipboard_parsing_enabled` | Boolean | Показывает кнопку "Вставить" |
| `debug_mode_enabled` | Boolean | Режим отладки — показывает диалог с результатом парсинга независимо от совпадения |

## 4. Алгоритм проверки баланса

```
Текущий баланс = SUM(INCOME) - SUM(EXPENSE) всех транзакций счёта - Сумма операции
Ожидаемый баланс = Баланс из SMS

Расхождение = Текущий баланс - Ожидаемый баланс
```

Если Расхождение != 0 → записать в комментарий транзакции:
```
Баланс расходится: ожидалось {Ожидаемый баланс}, получено {Текущий баланс}
```

## 5. Подбор категории

При наличии `shop` (магазин из сообщения):
1. Искать последнюю транзакцию **любого счёта** с таким же `shop`
2. Использовать её категорию
3. Если не найдено → категория = null

## 6. UI Flow

### 6.1 Кнопка "Вставить из буфера"

На главном экране "Операции" в TopAppBar (справа) отображается иконка `ContentPaste` когда `clipboard_parsing_enabled = true`.

При нажатии:
1. Читается буфер обмена (`coerceToText`)
2. Парсинг по regex для каждого счёта с `autoCreateEnabled`
3. Если совпал → `ClipboardDialog` → форма с предзаполненными данными
4. Если debug mode → `DebugClipboardDialog` всегда

### 6.2 Диалог подтверждения

- Показывается когда regex и cardMask совпали:
  - Оригинальный текст из буфера
  - Кнопка "Создать"
  - Кнопка "Отмена"

### 6.3 Форма операции (после "Создать")

- Открывается форма транзакции с предзаполненными полями:
  - Сумма (из именованной группы `amount`)
  - Магазин (из именованной группы `shop`, поле `shop`)
  - Счёт (найденный счёт)
  - Категория (подбор по магазину)
  - Комментарий (заполняется при расхождении баланса)
  - Дата = текущее время
  - source = CLIPBOARD
  - sourceData = оригинальный текст из буфера

- Транзакция ещё не сохранена — пользователь может отредактировать
- При сохранении → используется `SaveTransactionUseCase`

## 7. Обработка ошибок

| Ситуация | Результат |
|----------|-----------|
| Кнопка не видна (авто-создание выключено) | Ничего не делать |
| Буфер пустой | Ничего не делать |
| Regex не вернул все именованные группы | Ничего не делать |
| Маска карты не совпала | Ничего не делать |
| Нет счетов с autoCreateEnabled | Ничего не делать |

## 7.1 Режим отладки

При включённом `debug_mode_enabled` (Настройки → "Режим отладки"):

- Диалог отладки показывается **всегда** при нажатии кнопки "Вставить", если `clipboard_parsing_enabled` включён и буфер содержит текст
- Показывает результат парсинга для первого счёта с `autoCreateEnabled`:
  - Текст из буфера
  - Если regex совпал: счёт, amount, shop, cardMask (parsed vs account), balance
  - Если regex не совпал: причина (нет счетов, regex не совпал, cardMask не совпал)
- Кнопка "Перечитать" — заново прочитать буфер
- После закрытия диалога отладки:
  - Если совпал → показать `ClipboardDialog`
  - Если не совпал → ничего

## 8. Пример

### Входные данные

Текст в буфере:
```
*1026 оплата 339.98 р. MAGNIT MK GO Баланс 1441.81
```

Настройки счёта:
```kotlin
Account(
    id = "...",
    name = "Карта Сбер",
    autoCreateEnabled = true,
    cardMask = "1026",
    regexForText = """\*(?<cardMask>\d{4})\s+\S+\s+(?<amount>[\d]+(?:[.,]\d+)?)\s*р\.\s+(?<shop>[^.]+?)\s*\.\s+Баланс\s+(?<balance>[\d]+(?:[.,]\d+)?)"""
)
```

### Обработка

1. regexForText applied → groups: `amount="339.98"`, `shop="MAGNIT MK GO"`, `cardMask="1026"`, `balance="1441.81"`
2. Маска карты `1026` == `1026` (из группы `cardMask`) → совпала
3. Поиск категории: ищем последнюю транзакцию с shop = "MAGNIT MK GO"
   - Например, категория = "Продукты"
4. Показать диалог с предзаполненной категорией "Продукты"

### Сверка баланса

- Текущий баланс счёта: `1000.00` (из транзакций) - `339.98`
- Баланс из SMS: `1441.81`
- Сумма операции: `339.98`
- Ожидаемый баланс = `1441.81`

Расхождение = `660,02 - 1441.81 = -781,79`

Не 0 → комментарий: `Баланс расходится: ожидалось 660,02, получено 1441.81`

## 9. Требования к реализации

- SharedPreferences для глобальной настройки
- Расширение AccountEntity новыми полями (autoCreateEnabled, cardMask, regexForText)
- Расширение TransactionEntity (source, sourceData, shop)
- Regex валидация при сохранении счёта (проверка, что regex содержит именованные группы: amount, shop, cardMask, balance)
- Подбор категории по последней транзакции с таким же shop
- Приоритет: первый совпавший счёт, стоп после первого диалога
